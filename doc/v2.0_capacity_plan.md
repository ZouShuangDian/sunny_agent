# SunnyAgent v2.0 容量规划

> **版本**：v2.0 更开放、更稳定 | **周期**：Q2 2026（4-6月） | **更新日期**：2026-03-12
>
> **开发模式**：1 人 + AI 协作（人负责决策、架构、验收；AI 负责编码、测试、文档）

### 总日历时间

| 项目 | 天数 |
|------|------|
| Q2 总工作日（13 周） | 65 天 |
| 减：法定假期（清明、劳动节、端午） | -5 天 |
| 减：年假/病假预留 | -3 天 |
| **净工作日** | **57 天** |

### 有效开发时间

| 项目 | 占比 | 天数 |
|------|------|------|
| 净工作日 | 100% | 57 天 |
| 减：会议/沟通/BP 培训辅导 | -15% | -8.5 天 |
| 减：技术债务/Bug 修复/运维 | -15% | -8.5 天 |
| 减：未计划缓冲（紧急问题、需求变更） | -10% | -5.5 天 |
| **可用于功能开发的天数** | **60%** | **34.5 天** |

### 人 + AI 协作效率模型

> AI 不是简单的 2x 加速器，不同任务类型的 AI 杠杆率不同

| 任务类型 | AI 杠杆率 | 说明 |
|---------|----------|------|
| 后端 API / 业务逻辑 | 2.5x | AI 最擅长：CRUD、数据处理、算法实现，人只需审查和微调 |
| 前端 UI / 交互 | 1.5x | AI 能生成组件框架，但样式调整和交互细节仍需人工 |
| 架构设计 / 技术选型 | 1.2x | AI 辅助调研和方案对比，但决策依赖人的判断 |
| 第三方集成 / 调试 | 1.3x | API 对接、环境配置、兼容性问题 AI 帮助有限 |
| 测试编写 | 3.0x | AI 生成测试用例效率极高，人只需定义边界条件 |
| 文档 / 培训材料 | 3.0x | AI 可直接生成高质量初稿 |

**加权平均 AI 杠杆率**：约 **2.0x**（保守估计）

### 等效产能

```
实际可用天数：34.5 天
× AI 杠杆率：2.0x
= 等效产能：69 人天
```

## 四、风险因素与调整

### 不确定性较高的功能

| 功能 | 风险 | 可能超出 | 缓解方案 |
|------|------|---------|---------|
| RAG 知识库 | 向量检索准确率调优耗时 | +3~5天 | 先用成熟方案（LangChain + pgvector），不自研 |
| 桌面版应用 | Electron/Tauri 踩坑、跨平台适配 | +2~4天 | 先只支持单平台（macOS 或 Windows） |
| 数据 Connector | Iceberg 集成复杂度未知 | +2~3天 | 先做 DuckDB，Iceberg 降级为 Should |
| MCP 对接 | MCP 协议版本兼容性 | +1~2天 | 先对接最成熟的 2 个 MCP Server |

**最坏情况额外成本**：+8~14 天

### 调整后的现实评估

| 场景 | 乐观 | 正常 | 悲观 |
|------|------|------|------|
| AI 杠杆率 | 2.5x（86天） | 2.0x（69天） | 1.5x（52天） |
| Must Have 可行性 | 充裕 | 充裕 | 可完成但紧张 |
| Must + Should 可行性 | 可完成 | 勉强可完成 | 需砍 2-3 项 |

## 六、结论与建议

### 核心结论

1. **Must Have 5 项可以完成**，有约 31 等效人天的余量
2. **Must + Should 极限可行**，但几乎零缓冲，存在延期风险
3. **Could Have 大概率无法全部完成**，建议 Q2 只择优做 1-2 项

### 建议

| # | 建议 | 理由 |
|---|------|------|
| 1 | **桌面版先只支持单平台** | 跨平台适配是最大的不确定性，先 macOS 或 Windows 择一 |
| 2 | **Connector 先做 DuckDB，Iceberg 移到 v2.1** | 降低集成风险，DuckDB 已有成熟方案 |
| 3 | **数据分析 Agent 做基础版** | 先支持 CSV/Excel 分析，图表生成用现有库，不做复杂可视化 |
| 4 | **第 4 周末做第一次范围校准** | 根据实际 AI 杠杆率修正后续计划，避免末期才发现超载 |
| 5 | **BP 辅导时间不可压缩** | 这不是开发任务，是运营投入，需要预留固定时间槽 |

### 最终推荐发布范围

| 类别 | v2.0 发布（Q2） | v2.1 补充发布（Q3 初） |
|------|----------------|----------------------|
| **确定交付** | Must Have 全部 5 项 | Could Have 剩余项 |
| **尽力交付** | Should Have 4-6 项 | Iceberg Connector |
| **择优交付** | Could Have 1-2 项 | — |

### 团队配置方案

> 前提：每人 + AI 等效产能 69 天/季度，按角色专注度分配

**方案 A：最小团队（推荐）— 2 后端 + 1 前端**

| 角色 | 人数 | 等效产能 | 需要产能 | 利用率 |
|------|------|---------|---------|--------|
| 后端工程师（含 AI） | 2 | 138 天 | 49.5 + 12.5 = 62 天 | 45% |
| 前端工程师（含 AI） | 1 | 69 天 | 22.5 天 | 33% |
| **合计** | **3 人** | **207 天** | **85 天** | **41%** |

> 利用率偏低是因为：① 人员有学习曲线和项目熟悉成本；② 后端需承担集成/其他工作；③ 需预留缓冲应对风险。实际利用率建议不超过 70%，留 30% 给沟通协调、Code Review、技术债务。

**方案 B：紧凑团队 — 1 后端 + 1 全栈**

| 角色 | 人数 | 职责 | 等效产能 |
|------|------|------|---------|
| 后端工程师（含 AI） | 1 | RAG、Connector、MCP、质量追溯、数据分析 | 69 天 |
| 全栈工程师（含 AI） | 1 | Skills Hub、桌面版、Skill Creator、前端 UI | 69 天 |
| **合计** | **2 人** | | **138 天** |

> 138 天 vs 85 天需求，利用率 62%，有合理缓冲。但对人员能力要求更高，全栈需同时处理 Electron + React + 后端。

**方案 C：当前模式增强 — 1 人 + 1 前端外包/实习**

| 角色 | 人数 | 职责 | 等效产能 |
|------|------|------|---------|
| 你（后端 + AI） | 1 | 全部后端 + 架构 + BP 辅导 | 69 天 |
| 前端（外包/实习） | 1 | Skills Hub 前端、桌面版打包、UI 组件 | 40 天 |
| **合计** | **1+1 人** | | **109 天** |

> 109 天 vs 85 天，可行但紧张。前端外包需要你花时间对接需求和 Review。适合预算有限但需要加速的场景。

### 方案对比

| | 方案 A | 方案 B | 方案 C |
|---|--------|--------|--------|
| **人数** | 3 人 | 2 人 | 1+1 人 |
| **团队构成** | 2后端 + 1前端 | 1后端 + 1全栈 | 你 + 前端外包 |
| **等效产能** | 207 天 | 138 天 | 109 天 |
| **能否全部完成** | 充裕 | 可以 | 可以但紧张 |
| **缓冲余量** | 高（122天余量） | 中（53天余量） | 低（24天余量） |
| **协调成本** | 中（需求对齐+Review） | 低（2人默契） | 低（你主导） |
| **招聘难度** | 中 | 高（全栈难招） | 低（外包灵活） |
| **预算** | 高 | 中 | 低 |
| **推荐场景** | 要求按期全部交付 | 追求效率最大化 | 预算有限、快速启动 |

### 建议

> **如果只加 1 个人，优先加前端**。原因：
>
> 1. 后端工作 AI 杠杆率高（2.5x），你 + AI 可以覆盖大部分后端
> 2. 前端（Skills Hub UI + 桌面版打包）是当前最大瓶颈，AI 杠杆率只有 1.5x
> 3. 桌面版应用（Electron/Tauri）需要专门的前端经验，不适合 AI 独立完成
> 4. 前端外包/实习生的市场供给充足，启动快、成本可控
>
> **推荐方案 C**：你继续 1 人 + AI 主导后端和架构，补充 1 名前端处理 UI 和桌面版，Q2 全部 17 项功能可行。
