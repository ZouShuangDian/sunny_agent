# Feature Specification: 定时任务功能 (Scheduled Tasks)

**Feature Branch**: `scheduled-tasks`
**Created**: 2026-02-22
**Status**: Draft
**Input**: User description: "定时任务功能：支持在对话窗口和定时任务窗口发起定时任务；系统管理左边栏增加定时任务菜单，可启用/禁用、编辑、删除和立刻运行；通过图形化界面设置名称、提示词、计划（每天/每周/每月）和到期日期"


## Architecture Overview

定时任务采用 **APScheduler + 脚本文件** 的执行模式：

### 执行机制
- 定时任务通过 APScheduler（Python 调度库）调度执行，使用 PostgreSQL 作为 job store 实现持久化
- 每个任务关联一个脚本文件，脚本内容为用户的提示词/指令
- 脚本文件存储在用户专属目录下，按用户隔离

### 目录结构设计
```
data/
└── scheduled_tasks/
    └── {user_id}/              # 用户专属目录
        ├── scripts/            # 脚本文件目录
        │   ├── {task_id}.txt   # 任务脚本（提示词内容）
        │   └── ...
        └── logs/               # 执行日志目录
            ├── {task_id}/
            │   ├── {execution_id}.log
            │   └── ...
            └── ...
```

### 权限设计

| 角色 | 权限范围 |
|------|----------|
| 普通用户 | 只能访问自己的任务和脚本 |
| 管理员 | 可查看所有用户的任务，可全局启用/禁用定时任务功能 |

**权限规则**：
- 用户创建任务时，系统自动在用户目录下创建脚本文件
- 用户只能读写自己目录下的脚本文件
- 删除任务时，同步删除关联的脚本文件
- 用户被禁用或删除时，其所有任务和脚本文件一并清理

## UI Reference

基于原型设计，定时任务功能集成在现有"管理面板"弹窗中。

### 管理面板弹窗结构
现有管理面板弹窗左侧菜单：
- 用户管理
- 系统设置
- **定时任务**（新增）

点击不同菜单项，右侧显示对应的内容区。

### 定时任务列表（右侧内容区）
点击左侧"定时任务"菜单后，右侧显示任务列表：
- **页面标题**: 日历图标 + "定时任务"
- **操作按钮**: 右上角"+ 新建定时计划"按钮（蓝色按钮）
- **Tab切换**: "已定时" | "已完成" 两个标签页
- **列表列**: 标题、计划于（显示如"每天 09:00"、"每周一 10:00"）、状态（开关）、操作（立即运行、编辑、删除按钮）
- **空状态**: 当无任务时显示空状态提示和创建按钮

### "已完成"Tab（执行历史）
切换到"已完成"Tab 后，右侧内容区展示执行历史列表：
- **列表列**: 名称（对话标题，左对齐）、时间（执行时间，右对齐）
- **排序**: 按执行时间倒序
- **交互**: 点击某行 → 在左侧对话历史列表中定位并高亮对应的执行对话
- **数据来源**: 从 TaskExecution 关联的对话 session 中获取标题和时间

### 左侧对话历史（主页面侧边栏，非弹窗内）
管理面板是弹窗覆盖在主页面之上，而对话历史是主页面左侧边栏的固有组成部分。过滤功能作用于主页面的对话列表：
- **过滤按钮**: 对话历史标题栏右侧，漏斗图标
- **下拉菜单**: 点击过滤按钮弹出，提供两个选项
  - 全部对话（默认）
  - 定时任务（仅显示定时任务执行产生的对话）
- **过滤标签**: 选择非"全部"过滤后，列表顶部显示蓝色标签，含"定时任务" + 关闭按钮
- **定时任务标识**: 定时任务产生的对话项显示日历小图标，与手动对话视觉区分

### 添加/编辑定时任务弹窗
点击"+ 新建定时计划"或编辑按钮后，弹出居中模态弹窗：
- **弹窗类型**: 居中模态弹窗（Modal）
- **弹窗标题**: "添加定时任务" 或 "编辑定时任务"
- **表单字段**:
  - **标题**: 任务名称输入框（必填）
  - **提示词/脚本内容**: 多行文本输入框（必填），内容将保存为脚本文件
  - **计划**:
    - 重复类型下拉框：不重复、每天、每周、每月
    - 日期选择器：选择执行日期（不重复时显示）
    - 时间选择器：选择执行时刻
  - **到期日期**: 可选，日期选择器
- **操作按钮**: 取消、保存

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 创建定时任务 (Priority: P1)

用户希望设置一个定时任务，让系统在指定时间自动执行某个提示词/脚本（如每天早上生成市场分析报告）。用户通过管理面板创建任务，系统自动将提示词内容保存为脚本文件，并注册 APScheduler 调度。

**Why this priority**: 这是定时任务功能的核心能力，没有创建功能其他管理功能无意义。用户最基本的需求是能够设定自动化任务。

**Independent Test**: 可以通过创建一个简单的定时任务，验证任务保存成功、脚本文件生成、cron 任务注册正确。

**Acceptance Scenarios**:

1. **Given** 用户已登录并在管理面板中点击"定时任务"菜单项, **When** 用户点击"+ 新建定时计划"按钮并填写完整信息（标题、提示词、计划类型、日期、时间）后点击保存, **Then** 系统创建任务记录、生成脚本文件到用户目录、注册 APScheduler 调度，任务显示在"已定时"列表中
2. **Given** 用户在创建任务弹窗中, **When** 用户选择"不重复"计划类型, **Then** 系统显示单次执行的日期和时间选择器，任务执行一次后自动移至"已完成"并从 APScheduler 中移除
3. **Given** 用户在创建任务弹窗中, **When** 用户选择"每天"计划类型, **Then** 系统显示时间选择器，生成对应的 APScheduler cron 触发器表达式
4. **Given** 用户在创建任务弹窗中, **When** 用户选择"每周"计划类型, **Then** 系统显示星期选择器让用户选择具体执行日（可多选）和时间
5. **Given** 用户在创建任务弹窗中, **When** 用户选择"每月"计划类型, **Then** 系统显示日期选择器让用户选择每月执行的日期（1-31号，可多选）和时间
6. **Given** 用户在创建任务弹窗中, **When** 用户未填写必填项（标题或提示词）就点击保存, **Then** 系统显示验证错误提示，不保存任务

---

### User Story 2 - 管理定时任务列表 (Priority: P1)

用户需要查看所有已创建的定时任务，并能够对任务进行启用/禁用、编辑、删除和立即运行操作。任务列表通过Tab区分"已定时"（活跃任务）和"已完成"（一次性任务执行后或已过期的任务）。

**Why this priority**: 与创建功能同等重要，用户必须能够管理已创建的任务，这是基本的CRUD操作。

**Independent Test**: 可以通过查看任务列表，对已有任务执行各种操作，验证操作生效且 APScheduler 调度和脚本文件正确更新。

**Acceptance Scenarios**:

1. **Given** 用户在管理面板中点击"定时任务"菜单项, **When** 右侧内容区加载完成, **Then** 系统只显示当前用户的任务，默认显示"已定时"Tab
2. **Given** 用户在"已定时"Tab, **When** 用户点击"已完成"Tab, **Then** 系统显示所有已完成的一次性任务和已过期的重复任务
3. **Given** 任务列表中有一个已启用的任务, **When** 用户点击该任务的状态开关, **Then** 任务状态变为禁用，APScheduler 调度暂停，系统不再按计划执行该任务
4. **Given** 任务列表中有一个任务, **When** 用户点击编辑按钮, **Then** 系统打开编辑弹窗，显示当前任务配置，修改后同步更新脚本文件和 APScheduler 调度
5. **Given** 任务列表中有一个任务, **When** 用户点击删除按钮并确认, **Then** 系统删除任务记录、脚本文件和 APScheduler 调度
6. **Given** 任务列表中有一个任务, **When** 用户点击立即运行按钮, **Then** 系统立即执行该任务的脚本，并显示执行状态

### User Story 3 - 从对话窗口创建定时任务 (Priority: P2)

用户在与AI对话过程中，发现某个提示词很有价值需要定期执行，希望能直接从对话窗口将该提示词转换为定时任务，而无需离开当前对话。

**Why this priority**: 这是一个便捷功能，提升用户体验，但不是核心功能。用户仍可通过定时任务管理界面创建任务。

**Independent Test**: 可以在对话窗口中使用特定方式创建定时任务，验证任务成功创建并出现在定时任务管理列表中。

**Acceptance Scenarios**:

1. **Given** 用户正在对话窗口中, **When** 用户输入包含定时任务指令的消息（如"每天早上9点执行：分析今日新闻"）, **Then** 系统识别定时任务意图，弹出定时任务创建弹窗，预填提示词内容
2. **Given** 定时任务创建弹窗已弹出, **When** 用户确认或调整配置后点击保存, **Then** 系统创建定时任务和脚本文件，在对话中显示创建成功提示

---

### User Story 4 - 查看任务执行历史 (Priority: P2)

用户希望查看定时任务的历史执行记录，了解任务是否按时执行、执行结果如何。执行历史在管理面板"已完成"Tab中以列表形式展示（对话名称 + 执行时间），点击某条记录后在左侧对话历史中定位到对应的执行记录对话。

**Why this priority**: 执行历史是用户信任定时任务系统的关键，需要清晰展示每次执行的结果和可追溯性

**Independent Test**: 切换到"已完成"Tab 查看历史列表，点击某条记录后对话列表中高亮对应的执行对话。

**Acceptance Scenarios**:

1. **Given** 用户在定时任务管理界面, **When** 点击"已完成"Tab, **Then** 系统显示所有执行历史记录列表，每行包含对话名称（左侧）和执行时间（右侧），按时间倒序排列
2. **Given** 执行历史列表中有记录, **When** 用户点击某条记录, **Then** 系统在左侧对话历史列表中定位并高亮对应的执行对话记录
3. **Given** 用户点击了执行历史记录, **When** 对话历史中已定位到对应对话, **Then** 用户可以查看该次执行的完整 AI 响应内容

---

### User Story 5 - 对话历史中筛选定时任务 (Priority: P2)

用户在主界面的左侧对话历史列表中，需要快速筛选出定时任务产生的对话记录，与手动对话区分开来。

**Why this priority**: 定时任务执行产生的对话会混在手动对话中，筛选功能是基本的可用性需求

**Independent Test**: 点击对话历史区域的过滤按钮，选择"定时任务"后列表仅显示定时任务产生的对话。

**Acceptance Scenarios**:

1. **Given** 用户在主界面, **When** 点击对话历史左上方的过滤按钮, **Then** 弹出过滤下拉菜单，提供"全部对话"和"定时任务"两个选项
2. **Given** 过滤菜单已打开, **When** 用户选择"定时任务", **Then** 对话列表仅显示定时任务执行产生的对话，并在列表顶部显示过滤标签
3. **Given** 过滤标签已显示, **When** 用户点击标签上的关闭按钮, **Then** 恢复显示全部对话
4. **Given** 对话列表中有定时任务对话, **When** 查看定时任务对话项, **Then** 该项显示日历图标标识，与普通对话视觉区分

### Edge Cases

- 当"不重复"类型的任务执行完成后，系统应自动将其移至"已完成"Tab并从 APScheduler 中移除
- 当用户设置的到期日期已过时，系统应自动将任务移至"已完成"Tab并从 APScheduler 中移除
- 当用户删除正在执行的任务时，系统应提示"任务正在执行中，确定要删除吗？"，确认后终止执行并删除所有相关资源
- 当定时任务执行失败时，系统应记录失败原因到日志文件，并在 5 分钟后自动重试一次；若重试仍失败，等待下次计划时间
- 当用户选择每月31号执行，但该月没有31号时（如2月），系统应在该月最后一天执行
- 当用户同时有多个任务在同一时间执行时，系统最多并发执行 5 个任务，超出部分按创建时间顺序排队等待
- 当用户禁用账户或被删除时，其创建的所有定时任务、脚本文件和日志应一并清理
- 当脚本文件被意外删除时，系统应在执行时检测并记录错误，任务标记为异常状态
- 当任务执行超过 15 分钟时，系统应自动终止执行，记录超时失败状态，并在日志中标注超时原因

## Requirements *(mandatory)*

### Functional Requirements

**界面布局**
- **FR-001**: 在现有"管理面板"弹窗左侧菜单中，"系统设置"下方新增"定时任务"菜单项
- **FR-002**: 点击"定时任务"菜单项后，右侧内容区显示任务列表，提供"已定时"和"已完成"两个Tab切换
- **FR-003**: 任务列表必须显示以下列：标题、计划于、状态（开关）、操作按钮（立即运行、编辑、删除）
- **FR-004**: 任务列表右上角必须提供"+ 新建定时计划"按钮

**创建任务**
- **FR-005**: 系统必须支持创建定时任务，包含以下必填字段：标题、提示词/脚本内容
- **FR-006**: 系统必须支持四种计划类型：不重复（一次性）、每天、每周、每月
- **FR-007**: 系统必须支持设置任务执行日期和时间，并转换为 APScheduler cron 触发器表达式
- **FR-008**: 系统必须支持设置任务到期日期（可选），到期后任务移至"已完成"
- **FR-009**: 对于每周计划，系统必须支持选择一周中的多个执行日（周一至周日）
- **FR-010**: 对于每月计划，系统必须支持选择一月中的多个执行日期（1-31号）

**脚本文件管理**
- **FR-011**: 创建任务时，系统必须将提示词内容保存为脚本文件，存储在用户专属目录下
- **FR-012**: 编辑任务时，系统必须同步更新对应的脚本文件内容
- **FR-013**: 删除任务时，系统必须同步删除对应的脚本文件和执行日志
- **FR-014**: 脚本文件命名规则：`{task_id}.txt`，存储路径：`data/scheduled_tasks/{user_id}/scripts/`

**任务管理**
- **FR-015**: 系统必须在任务列表中显示任务状态开关，允许用户启用或禁用任务
- **FR-016**: 启用任务时注册 APScheduler 调度任务，禁用任务时暂停调度任务
- **FR-017**: 系统必须支持通过编辑按钮编辑已创建的定时任务的所有配置项
- **FR-018**: 系统必须支持通过删除按钮删除定时任务，删除前需用户确认
- **FR-019**: 系统必须支持"立即运行"功能，手动触发脚本执行
- **FR-020**: "不重复"类型任务执行完成后必须自动移至"已完成"Tab

**任务执行**
- **FR-021**: 系统必须通过 APScheduler 按照设定的计划自动执行启用状态的定时任务
- **FR-022**: 执行时读取用户目录下的脚本文件内容，作为提示词发送给AI处理
- **FR-023**: 系统必须记录每次任务执行的历史，包含执行时间、状态和结果
- **FR-024**: 执行日志必须保存到用户目录：`data/scheduled_tasks/{user_id}/logs/{task_id}/`
- **FR-025**: 系统必须将定时任务执行的结果保存为对话记录，用户可查看完整的AI响应
- **FR-025a**: 系统必须通过 Langfuse 追踪每次任务执行，与现有 AIME 追踪保持一致，支持执行链路分析和成本统计

**执行历史查看**
- **FR-031**: "已完成"Tab 必须展示任务执行历史列表，每行显示对话名称（左）和执行时间（右），按时间倒序排列
- **FR-032**: 点击执行历史中的某条记录，必须在左侧对话历史列表中定位并高亮对应的执行对话
- **FR-033**: 执行历史记录必须关联到对应的对话 session，支持跳转查看完整 AI 响应

**对话历史过滤**
- **FR-034**: 左侧对话历史区域左上方必须提供过滤按钮
- **FR-035**: 过滤下拉菜单必须支持两种筛选："全部对话"（默认）和"定时任务"
- **FR-036**: 选择"定时任务"过滤后，对话列表仅显示定时任务执行产生的对话，并在顶部显示过滤标签
- **FR-037**: 过滤标签必须支持一键清除，恢复显示全部对话
- **FR-038**: 定时任务产生的对话在列表中必须显示日历图标标识，与普通对话视觉区分

**权限与隔离**
- **FR-026**: 定时任务必须与创建它的用户关联，用户只能查看和管理自己的任务
- **FR-027**: 用户只能访问自己目录下的脚本文件和执行日志
- **FR-028**: 管理员可查看所有用户的任务列表（只读），可全局启用/禁用定时任务功能
- **FR-029**: 用户被禁用或删除时，系统必须清理该用户的所有任务、脚本文件和日志
- **FR-030**: 系统必须支持从对话窗口创建定时任务，通过识别用户的定时任务意图触发

### Key Entities

- **ScheduledTask（定时任务）**: 定时任务的核心实体
  - 任务ID、标题、计划类型（once/daily/weekly/monthly）
  - 执行时间、执行日期列表、到期日期（可选）
  - APScheduler cron 触发器表达式、启用状态
  - 脚本文件路径、创建用户ID、创建时间、更新时间

- **TaskScript（任务脚本）**: 脚本文件元数据
  - 脚本ID、关联任务ID、文件路径
  - 内容哈希（用于检测变更）、创建时间、更新时间

- **TaskExecution（任务执行记录）**: 记录每次任务执行
  - 执行ID、所属任务ID、执行时间
  - 执行状态（pending/running/success/failed）
  - 日志文件路径、关联的对话 session_id（用于"已完成"列表跳转）、执行耗时
  - 对话名称（冗余存储，用于"已完成"列表展示，取自对话 session 标题）

## Assumptions

- 用户时区假设为服务器时区，暂不支持用户自定义时区
- 定时任务执行时使用用户的默认AI配置（如使用的模型）
- 系统启动时自动恢复所有启用状态的定时任务到 APScheduler 调度
- 任务执行生成的对话独立于用户的日常对话，归类为"定时任务对话"（对话 session 表中通过 source 字段区分：`manual` / `scheduled`）
- 管理员可以在系统层面设置定时任务功能的全局开关
- 脚本文件使用 UTF-8 编码存储
- 单个脚本文件大小限制为 64KB
- 单次任务执行超时限制为 15 分钟，超时后自动终止并记录失败状态
- 系统最多允许 5 个任务并发执行，超出的任务进入等待队列
- 任务执行失败后自动重试 1 次（5 分钟延迟），重试失败则等待下次计划时间
- 任务执行通过 Langfuse 追踪，复用现有 AIME 的追踪配置和基础设施
- 执行日志保留最近 90 天，超期自动清理

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可以在2分钟内完成一个定时任务的创建流程
- **SC-002**: 定时任务按计划执行的准确率达到99%（允许1分钟内的误差）
- **SC-003**: 任务列表页面在3秒内完成加载（包含100个任务的情况下）
- **SC-004**: 95%的用户首次使用时能够成功创建定时任务，无需查阅帮助文档
- **SC-005**: "立即运行"操作在5秒内开始执行任务
- **SC-006**: 任务执行历史保留最近90天的记录，支持分页查看
- **SC-007**: 用户数据完全隔离，任何用户无法访问其他用户的脚本文件或日志
