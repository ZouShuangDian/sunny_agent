## Context

### 背景

Sunny Agent 当前仅提供 Web 聊天界面，用户需要通过浏览器访问。企业用户（特别是已部署飞书/钉钉的组织）期望能在 IM 工具内直接交互，减少平台切换成本。

### 当前架构

```
用户 → Web 界面 → FastAPI → Agent Pipeline → LLM
```

### 目标架构

```
飞书用户 → 飞书机器人 → feishu-webhook → Redis → Worker → Agent Pipeline → LLM
                                          ↓
                                    流式卡片实时更新
```

### 参考实现

借鉴 OpenClaw 飞书集成方案：
- WebSocket 连接改为 Webhook + 队列消费
- BlockStreaming 策略用于累积和分块回复
- 流式卡片（CardKit）实现打字机效果

## Goals / Non-Goals

**Goals:**

1. 实现飞书消息接收与异步处理（Webhook → Redis → Worker）
2. 支持访问控制（dm_policy/group_policy/白名单）
3. 实现 BlockStreaming 回复策略（OpenClaw 风格）
4. 支持媒体文件下载与长期保存
5. 实现流式卡片实时更新（50ms/次）
6. 提供完整审计日志与监控

**Non-Goals:**

1. 不支持飞书主动推送消息（仅被动接收和回复）
2. 不支持复杂审批流程（仅白名单控制）
3. 不实现飞书小程序/H5 页面
4. 不修改现有 Web 聊天功能

## Decisions

### 1. 架构模式：Webhook + 队列消费（而非 WebSocket）

**决策:** 采用 Webhook + Redis List + Worker 消费，而非 OpenClaw 的 WebSocket 模式。

**理由:**
- **水平扩展性**: Worker 可以水平扩展，不受 WebSocket 连接数限制
- **解耦**: Webhook 服务与 Worker 独立部署，故障隔离
- **可靠性**: Redis 队列提供持久化，消息不丢失
- **简单性**: 无需处理 WebSocket 心跳、重连等复杂逻辑

**替代方案:**
- WebSocket 直连：实时性好，但扩展性受限，代码复杂度高
- HTTP 长轮询：复杂度介于两者之间，但实时性不如 WebSocket

### 2. 队列架构：独立 ARQ 队列（方案A优化）

**决策:** 飞书消息使用独立的 ARQ 队列，不与定时任务队列共享。通过独立的 Worker 进程处理飞书消息，实现资源隔离和独立扩缩容。

**数据流:**
```
feishu-webhook → Redis List → message_transfer_loop (BRPOPLPUSH 长驻任务) → ARQ Queue (feishu) → process_feishu_message (ARQ Task)
                                                                      ↑
                                                           独立的 Feishu Worker (arq:feishu:queue)
```

**架构设计:**
- **独立队列**: `arq:feishu:queue`，与 `arq:queue`（定时任务）完全分离
- **独立 Worker**: 单独的 Feishu Worker 进程，专门消费飞书消息队列
- **共享 Redis**: 使用同一个 Redis 实例，但不同队列名称
- **统一技术栈**: 仍使用 ARQ，获得重试、监控、死信队列等能力

**理由:**
- **资源隔离**: 飞书消息处理和定时任务互不干扰，避免资源竞争
- **独立扩缩容**: 可根据飞书消息量独立扩容 Feishu Worker，不影响定时任务
- **故障隔离**: 飞书 Worker 崩溃不会影响定时任务执行
- **灵活配置**: 可为飞书 Worker 配置不同的并发数、超时时间等参数
- **内置可靠性**: 仍获得 ARQ 的重试、超时、死信队列、进度追踪等能力

**权衡:**
- **部署复杂度**: 需要部署两个 Worker 进程（定时任务 Worker + Feishu Worker）
- **资源占用**: 额外的进程开销
- **延迟增加**: 消息转移间隔（1秒）+ Debounce 等待（2秒），总延迟约 3-4 秒（可接受）

**队列命名:**
- 定时任务: `arq:queue`（默认）
- 飞书消息: `arq:feishu:queue`（独立）

**替代方案对比:**
- **统一队列（原方案A）**: 共享 Worker，部署简单但资源竞争、无法独立扩缩容
- **方案B（独立飞书 Worker 非ARQ）**: 完全独立实现，失去 ARQ 能力
- **Redis Stream**: 功能强大但增加复杂度，与 ARQ 不兼容

### 3. 访问控制策略：白名单 + Policy（去掉配对模式）

**决策:** 支持 dm_policy (open/allowlist/disabled) 和 group_policy，**去掉** OpenClaw 的 pairing 配对模式。

**理由:**
- **简单性**: 配对模式需要复杂的审批流程和状态管理
- **企业场景**: 企业用户通常有明确的员工清单，适合白名单模式
- **减少复杂度**: 去掉 pairing 后，用户映射逻辑更简单

**替代方案:**
- 保留 pairing 模式：适合面向公众的场景，但增加复杂度
- OAuth 集成：需要飞书 OAuth，实现成本高

### 4. BlockStreaming 策略：借鉴 OpenClaw + 增强

**决策:** 采用 OpenClaw 的 BlockStreaming 策略，并增加 **段落感知刷新 (flushOnEnqueue)**：

**基础参数：**
- `min_chars` (800): 累积最小字符数
- `max_chars` (1200): 强制 flush 阈值
- `idle_ms` (1000): 空闲等待时间

**增强功能：**
- `flush_on_enqueue` (true): 检测到段落边界时立即 flush
- `paragraph_aware` (true): 段落感知模式，保持段落完整性

**段落边界检测规则：**
- 双换行符 (`\n\n`): 段落分隔
- 列表标记 (`\n- `, `\n* `, `\n1. `): 列表项边界
- 代码块标记 (``\n`): 代码块边界
- 标点符号 (`.\n`, `?\n`, `!\n`): 句子结束

**理由:**
- **验证有效**: OpenClaw 已在生产环境验证
- **平衡体验**: 在实时性和消息完整性之间取得平衡
- **段落完整性**: 避免在句子或段落中间截断，提升结构化内容（列表、代码块）的阅读体验
- **可配置**: 参数可通过配置表调整

**替代方案:**
- 逐 token 发送：实时性最好，但可能产生过多消息
- 整段发送：简单，但用户等待时间长
- 纯字符/时间触发：简单，但可能在奇怪位置截断内容

### 5. 人机延迟：模拟人类回复节奏（借鉴 OpenClaw）

**决策:** 在 AI 生成完成后、发送回复前增加随机延迟，模拟人类打字停顿。

**核心设计:**
- **延迟范围**: 默认 500-1500ms（比 OpenClaw 的 800-2500ms 稍快，更适合企业场景）
- **随机算法**: uniform random，在 [min_ms, max_ms] 范围内随机选择
- **应用时机**: AI 生成完成 → 等待随机延迟 → 开始发送回复
- **可配置**: 支持在配置表中针对不同应用/群组/用户单独调整

**配置参数:**
```json
{
  "human_like_delay": {
    "enabled": true,
    "min_ms": 500,
    "max_ms": 1500
  }
}
```

**理由:**
- **体验自然**: 避免 AI 回复过快显得机械，提升用户体验
- **企业适配**: 500-1500ms 比 OpenClaw 的 800-2500ms 更紧凑，符合企业效率要求
- **可关闭**: 支持通过配置禁用，适合对延迟敏感的场景
- **细粒度控制**: 支持应用级、群组级、用户级配置

**替代方案:**
- 固定延迟：简单，但不够自然
- 无延迟： fastest，但显得像机器人
- 打字机延迟：复杂，需在前端实现

### 6. Inbound Debounce：Redis + 双阶段策略（借鉴 CoPaw）

**决策:** 采用 Redis 持久化存储 + 双阶段 Debounce 策略（Time Debounce + No-Text Debounce）+ should_debounce 钩子。

**核心设计:**
- **存储**: Redis（支持分布式，Worker 重启不丢失）
- **状态机**: idle → buffering → processing → idle
- **Time Debounce**: 时间防抖（默认 2 秒），收到新消息重置定时器
- **No-Text Debounce**: 无文本消息缓冲（默认 3 秒），等待文字说明
- **should_debounce 钩子**: 可配置回调，精细控制是否 debounce
- **批量 Drain**: 批量消费同一 session 的消息（默认最多 10 条）

**理由:**
- **可靠性**: Redis 持久化，支持 Worker 水平扩展和故障恢复
- **智能合并**: 双阶段策略学习 CoPaw，适合移动端"先图后文"习惯
- **灵活性**: should_debounce 钩子学习 OpenClaw，支持业务定制
- **性能**: 批量 drain 优化，减少 Redis 调用次数

**配置参数:**
```json
{
  "debounce": {
    "debounce_wait_seconds": 2.0,        // 时间防抖等待期
    "no_text_debounce": {
      "enabled": true,                    // 是否启用无文本防抖
      "max_wait_seconds": 3.0            // 无文本消息最长等待
    },
    "max_batch_size": 10,                // 批量消费最大条数
    "should_debounce_hook": null         // 自定义钩子路径
  }
}
```

**替代方案:**
- **纯内存**: 如 OpenClaw（简单但无法扩展）
- **纯时间防抖**: 去掉 No-Text Debounce（更简单但不够智能）
- **无钩子**: 固定逻辑（不够灵活）

### 6. 流式卡片：第一块流式，后续块普通消息

**决策:** 超长文本（>2000字符）分块时，第一块用流式卡片，后续块用普通文本消息。

**理由:**
- **用户体验**: 第一块展示打字机效果，让用户感知到 AI 正在工作
- **技术限制**: 飞书流式卡片只能更新同一张卡片，无法分多张卡片
- **简洁性**: 后续块用普通消息，避免卡片过多占用聊天空间

**替代方案:**
- 全部用流式卡片：无法分多张卡片，超长文本会截断
- 全部用普通消息：失去打字机效果

### 6. 媒体存储：长期保存（而非临时）

**决策:** 媒体文件长期保存到本地存储，创建数据库记录。

**理由:**
- **可复用性**: 媒体文件可在多个会话中引用
- **审计需求**: 企业场景可能需要追溯历史媒体
- **合规性**: 满足数据留存要求

**替代方案:**
- 临时存储：节省空间，但无法复用和审计
- 对象存储（S3/OSS）：扩展性好，但增加依赖

### 7. Token 管理：Worker 自行管理（而非集中服务）

**决策:** Worker 直接调用飞书 API 获取 token，缓存到 Redis。

**理由:**
- **简单性**: 无需额外的 Token 服务
- **可靠性**: Redis 缓存提供高可用
- **解耦**: feishu-webhook 仅接收消息，不管理凭证

**替代方案:**
- 集中 Token 服务：适合多服务共享 token，但增加复杂度
- feishu-webhook 管理：增加 webhook 服务职责，不利于解耦

## Risks / Trade-offs

### 风险 1: Debounce 延迟影响用户体验

**风险:** Inbound Debounce 引入 2 秒等待期，用户可能感知到延迟。

**缓解措施:**
- 配置化：等待时间可通过配置表调整（1-5 秒）
- 优化场景：仅对连续消息启用 debounce，单条消息立即处理
- 监控：记录 debounce 耗时，持续优化

### 风险 2: BlockStreaming 累积延迟

**风险:** BlockStreaming 可能引入 1-3 秒额外延迟（等待累积到 min_chars）。

**缓解措施:**
- 参数调优：根据实际场景调整 min_chars 和 idle_ms
- 快速首包：优先发送流式卡片"思考中"状态，缓解等待焦虑
- 段落感知刷新：flush_on_enqueue 在段落边界立即发送，减少等待时间
- 可禁用：支持通过配置关闭 block streaming，直接整段发送

### 风险 3: 段落感知刷新边界检测准确性

**风险:** 段落边界检测可能在特殊格式文本（如代码块、表格）上出现误判，导致 flush 位置不理想。

**缓解措施:**
- 可配置边界模式：支持自定义 boundary_patterns，适应不同场景
- 降级策略：检测失败时回退到时间/字符数触发
- 测试覆盖：针对 Markdown、代码块、表格等格式进行充分测试
- 监控：记录 flush 位置，分析异常模式

### 风险 4: 人机延迟与效率权衡

**风险:** 人机延迟引入 500-1500ms 额外延迟，企业用户可能觉得回复不够迅速。

**缓解措施:**
- 可配置：支持完全禁用人机延迟，适合效率优先场景
- 范围可调：支持调整延迟范围（如改为 200-500ms）
- 场景识别：对系统命令、快捷操作等可自动跳过延迟
- 监控：记录延迟耗时，收集用户反馈持续优化
- A/B 测试：不同用户群体使用不同延迟配置，对比满意度

### 风险 5: 飞书 API 限流

**风险:** 高频调用飞书 API（token 获取、消息发送）可能触发限流。

**缓解措施:**
- Token 缓存：TTL 7000s，减少 token 获取频率
- 指数退避：限流时自动退避重试（1s, 2s, 4s, 8s）
- 监控告警：记录限流事件，触发告警

### 风险 5: 媒体文件存储占用

**风险:** 长期保存媒体文件可能导致存储空间快速增长。

**缓解措施:**
- 大小限制：30MB/文件，拒绝超大文件
- 清理策略：支持配置保留策略（如保留 30 天）
- 存储监控：监控存储使用情况，触发告警

### 风险 6: Worker 单点故障

**风险:** Worker 崩溃可能导致消息积压或丢失。

**缓解措施:**
- 多实例部署：Worker 可水平扩展，多个实例消费同一队列
- 队列持久化：Redis 持久化配置，防止数据丢失
- 健康检查：Worker 健康检查，异常时自动重启
- 监控告警：队列长度监控，积压时告警

### 风险 7: 并发消息处理冲突

**风险:** 同一用户快速发送多条消息，可能导致回复乱序或覆盖。

**缓解措施:**
- 会话锁：Redis 分布式锁，确保同一会话串行处理
- 状态机：明确的状态流转（idle → buffering → processing → idle）
- 序列号：流式卡片更新使用递增序列号，确保顺序

## Migration Plan

### 阶段 1: 基础设施准备

1. **数据库迁移**
   ```sql
   -- 执行 migrations 创建 5 张新表
   python -m alembic upgrade head
   ```

2. **Redis 配置**
   - 确认 Redis 可访问
   - 配置持久化策略

3. **飞书应用创建**
   - 在飞书开放平台创建应用
   - 配置事件订阅（Webhook URL）
   - 记录 app_id, app_secret, encrypt_key

### 阶段 2: 服务部署

1. **部署 feishu-webhook 服务**
   ```bash
   docker-compose up -d feishu-webhook
   ```

2. **配置环境变量**
   ```bash
   # .env
   FEISHU_WEBHOOK_PORT=8001
   FEISHU_APP_ID=cli_xxx
   FEISHU_APP_SECRET=xxx
   FEISHU_ENCRYPT_KEY=xxx
   REDIS_URL=redis://localhost:6379/0
   ```

3. **验证 Webhook 连通性**
   - 飞书平台发送测试事件
   - 验证消息到达 Redis 队列

### 阶段 3: Worker 更新

1. **更新 Worker 代码**
   ```bash
   git pull origin main
   pip install -r requirements.txt  # 新增 lark-oapi
   ```

2. **重启 Worker**
   ```bash
   # 滚动重启，确保零停机
   docker-compose restart worker
   ```

3. **验证消费**
   - 发送测试消息到飞书
   - 验证 Worker 消费并处理

### 阶段 4: 配置与测试

1. **初始化访问控制配置**
   ```sql
   INSERT INTO feishu_access_config (...)
   ```

2. **白名单配置**
   ```sql
   INSERT INTO feishu_user_bindings (...)
   ```

3. **端到端测试**
   - 私聊测试
   - 群聊 @测试
   - 媒体文件测试
   - 超长文本测试

### 阶段 5: 灰度与监控

1. **灰度发布**
   - 小范围用户测试
   - 收集反馈

2. **监控配置**
   - 队列长度监控
   - 处理耗时监控
   - 错误率监控

3. **全量发布**
   - 开放给所有用户

### 回滚策略

**回滚触发条件:**
- 错误率 > 5%
- 队列积压 > 1000 条
- 飞书 API 持续限流

**回滚步骤:**
1. 停止 feishu-webhook 服务（停止接收新消息）
2. 等待 Worker 处理完队列中现有消息
3. 回滚 Worker 代码到上一版本
4. 重启 Worker
5. 通知用户服务暂时不可用
